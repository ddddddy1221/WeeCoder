# WeeCoder 业务交付 Pipeline 设计

日期：2026-06-30

## 目标

WeeCoder 需要一条清晰、可解释、可运营的业务开发流转 Pipeline。它不是一张静态流程表，而应该成为管理后台的主工作模型：每个项目都能看到阶段、负责人、自动化程度、阶段闸口、交付产物、风险、阻塞和交接记录。

当前系统已经具备一个可用的核心流程：

1. 项目入口
2. 项目经理需求
3. PRD 审批
4. 架构与数据设计
5. 运维需求
6. 自动开发
7. 代码 / 安全 / 性能审查
8. 测试
9. 缺陷回归
10. 最终验收

下一步不是推翻这套流程，而是把它升级成更完整的交付视图，补齐 UI / 交互设计、ERD、测试用例、黑盒测试、白盒审查、运维脚本和部署记录。

## 设计原则

每个阶段必须回答五个问题：

- 谁负责这个阶段？
- AI 可以自动化什么？
- 哪些内容必须由人确认？
- 什么产物能证明阶段完成？
- 什么闸口会阻止进入下一阶段？

自动化不是简单的“人工 / 自动”二分，而应该分为四种模式：

- 人工负责：人做最终内容判断和责任确认。
- AI 辅助：AI 生成初稿、检查或建议，人确认。
- AI 主导：AI 负责主要执行，人做审查闸口。
- 系统闸口：平台根据缺失证据、风险或状态自动阻止流转。

## 最新推荐 Pipeline

| 顺序 | 阶段 | 工作模式 | 主负责人 | 必要产物 | 阶段闸口 |
| --- | --- | --- | --- | --- | --- |
| 1 | 需求提交 / BRD | 人工负责 | 总负责人 / 业务负责人 | 业务背景、目标、负责人、优先级 | 项目目标和初始范围已说明 |
| 2 | 需求澄清 | AI 辅助 | 项目经理 | 结构化需求答案、缺失项清单 | 关键问题已补齐到可生成 PRD |
| 3 | PRD 生成与审批 | AI 辅助 + 系统闸口 | 项目经理 + 总负责人 | PRD 版本、变更影响、审批意见 | 需求质检通过，PRD 版本冻结 |
| 4 | UI / 交互设计 | 人工负责，后续 AI 辅助 | 产品 / 设计 | 页面流程、交互说明、线框图或截图 | 核心用户路径可评审 |
| 5 | ERD / 技术设计 | AI 主导 + 技术负责人审批 | 技术负责人 | ERD、接口契约、模块边界、任务拆分 | 数据模型和服务契约已审批 |
| 6 | 运维需求 | AI 辅助 | 运维 | 运行环境、环境变量、密钥清单、网络和部署约束 | 开发前置运行条件已确认 |
| 7 | 代码编写与集成 | AI 主导 | AI 开发 + 技术负责人 | 变更包、提交记录、自测结果、实现说明 | 构建 / 测试通过，变更包可审查 |
| 8 | 测试用例编写 | AI 辅助 | 测试 | 功能用例、边界用例、回归用例、验收指标 | 用例覆盖 PRD 和 Review 交接重点 |
| 9 | 黑盒测试 | 人工负责 + 自动化辅助 | 测试 | 执行记录、缺陷、截图 / 日志、通过率 | 关键路径通过或缺陷已路由 |
| 10 | 白盒测试 / 安全 / 质量审查 | AI 主导 + 技术负责人审批 | 技术负责人 | 代码审查报告、安全问题、性能问题 | 无阻塞级代码 / 安全 / 性能风险 |
| 11 | 运维脚本与发布准备 | AI 辅助 | 运维 + AI 开发 | 启停脚本、部署说明、回滚说明 | 部署包可重复执行 |
| 12 | 运维部署 | 人工负责 | 运维 | 部署记录、环境、版本、操作人 | 服务已部署或部署决策已记录 |
| 13 | 最终验收 | 人工负责 + 系统闸口 | 总负责人 | 验收包、签收结论、遗留风险 | 负责人签收或退回指定阶段 |

## 前端分组方式

前端不应该把 13 个阶段等权铺满页面，否则会继续显得冗长。建议把完整 Pipeline 分成五个业务带：

1. 需求带
   - 需求提交 / BRD
   - 需求澄清
   - PRD 生成与审批

2. 设计带
   - UI / 交互设计
   - ERD / 技术设计
   - 运维需求

3. 构建带
   - 代码编写与集成
   - 运维脚本与发布准备

4. 验证带
   - 测试用例编写
   - 黑盒测试
   - 白盒测试 / 安全 / 质量审查
   - 必要时进入缺陷回归

5. 发布带
   - 运维部署
   - 最终验收

这个分组能让页面保持清晰，同时覆盖完整交付生命周期。

## 与当前代码的映射

当前 `src/shared/workflow.js` 里的阶段模型先作为稳定工作流引擎，不建议一次性重写。新的 Pipeline 应该先作为展示层和元数据层叠加上去，再逐步改造底层阶段。

| 当前阶段 | 新 Pipeline 覆盖内容 | 处理建议 |
| --- | --- | --- |
| intake | 需求提交 / BRD | 保持第一道项目入口闸口 |
| pm-requirements | 需求澄清 | 继续使用需求问题和需求质检 |
| prd-approval | PRD 生成与审批 | 继续使用 PRD 版本和变更过期阻断 |
| architecture | ERD / 技术设计，部分 UI / 交互设计 | 增加 UI / 交互和 ERD 的显式元数据 |
| ops-requirements | 运维需求 | 保持为开发前置条件 |
| development | 代码编写与集成、运维脚本 | 在 UI 上区分应用代码和运维脚本 |
| review | 白盒测试 / 安全 / 质量审查 | 保持代码、安全、性能审查能力 |
| qa | 测试用例编写、黑盒测试 | 在 UI 上拆成用例设计和测试执行 |
| defect-loop | 缺陷回归 | 保持为条件回流，不作为普通线性阶段 |
| acceptance | 运维部署、最终验收 | 在最终签收前补部署记录 |

## 推荐后台布局

项目详情页建议改成这个结构：

- 顶部总览：项目名称、当前业务带、当前阶段、健康度、阻塞数、负责人。
- 左侧 Pipeline：按业务带分组展示紧凑阶段轨道。
- 中间工作区：当前阶段任务、必要产物、操作按钮、最近输出。
- 右侧控制面板：阻塞、待确认、风险登记、角色交接。
- 底部抽屉或 Tab：产物历史、审计日志、PRD 版本、Review 报告。

这样可以避免一个页面里堆满所有模块，也更适合负责人快速看进度。

## 阶段卡片数据契约

每个阶段卡片都应该使用一致的数据结构：

- `id`：阶段唯一标识
- `name`：阶段名称
- `band`：所属业务带
- `ownerRole`：负责人角色
- `operatingMode`：工作模式
- `automationStatus`：自动化能力状态
- `humanGate`：人工确认点
- `requiredArtifacts`：必要产物
- `entryCriteria`：进入条件
- `exitCriteria`：完成条件
- `risks`：阶段风险
- `blockers`：当前阻塞
- `nextAction`：下一步动作

这样 UI 可以用同一套卡片逻辑渲染不同阶段。

## 闸口与回流规则

主流程需要遵守这些规则：

- PRD 审批后，如果需求变化，下游开发和测试必须标记为过期，直到重新生成或重新审批 PRD。
- 开发不能在 PRD、技术设计和运维前置条件缺失时启动。
- Review 不能在开发变更包未就绪时启动。
- QA 不能在缺少测试证据或 Review 交接时完成。
- QA 发现缺陷后可以回流开发，但修复后必须重新经过 Review 和 QA 回归。
- 部署必须是人工负责的运营动作，即使部署脚本由 AI 生成。
- 最终验收必须同时具备 PRD、实现摘要、Review 报告、QA 证据、运维交接和遗留风险。

## 实施路线

下一步实施建议分六步：

1. 增加 Pipeline 元数据层，不破坏当前 workflow 引擎。
2. 按五个业务带重构项目详情页。
3. 把 QA 区域拆成测试用例、黑盒执行、测试证据和缺陷路由。
4. 在设计带里增加 UI / 交互和 ERD 面板。
5. 在最终验收前增加运维脚本和部署记录面板。
6. 项目级流程稳定后，再做组织级 Pipeline 统计和跨项目看板。

## 下一版不做的事

- 不一次性替换当前 workflow 引擎。
- 不自动执行真实部署。
- 不做完整的 Figma 类设计工具。
- 不把 YOLO 专属字段强行放进所有项目。
- 不在这次 Pipeline 重构里做多租户计费、SLA 或商业化权限体系。

## 成功标准

下一版 Pipeline 优化成功的标准：

- 总负责人 10 秒内能看懂每个项目卡在哪个交付阶段。
- 项目经理不用读完整 PRD，也能看到缺失需求和 PRD 版本风险。
- 技术负责人能看到开发是否允许启动，以及阻塞原因。
- 测试能区分测试用例设计、测试执行和缺陷回流。
- 运维能看到运行前置条件、脚本、部署说明和交接状态。
- 总负责人能看到最终验收前还缺什么。
